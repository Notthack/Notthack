# Phase 5 — Pilot Path

## Fixed Pilot Hypothesis

`Pilot with a university student affairs office running a hardship meal support program across 1–2 campus cafeterias.`

## Why This Pilot Is Plausible

- the actor set is small and identifiable
- the voucher redemption flow already exists in some form
- the institution has direct incentive to reduce misuse and reconciliation friction
- the user group is accessible and understandable
- it avoids needing national government integration for the first test

## Pilot Partner Guess

Primary:

- university student affairs or welfare office

Secondary:

- approved campus food vendors
- university finance / internal audit office

## What A Pilot Would Actually Test

- can a QR credential be redeemed without wallet UX?
- can duplicate redemption be blocked in practice?
- can finance staff audit the shared redemption history faster?
- can manual override be kept rare and accountable?

## MVP Checklist

- one issuer panel
- one student QR pass
- one merchant scan-and-redeem screen
- one duplicate redemption block
- one auditor checkpoint view
- one override / appeal policy note

## Non-MVP Backlog

- multi-campus federation
- integration with student information systems
- offline redemption mode
- accessibility localization pack
- role-based analytics
- broader hardship program categories

## Deployment Risk Notes

- real eligibility policy is sensitive and should stay simulated in the hackathon demo
- merchant training and override policy matter as much as software
- appeal handling must remain human-controlled

## Phase 5 Exit Check

- one deployment context fixed: yes
- thesis frozen: yes
- minimum actor set defined: yes
- on-chain / off-chain split frozen: yes
- 90–120 second demo path defined: yes
- duplicate redemption failure case defined: yes
- normal DB rebuttal prepared: yes
- pilot path plausible: yes
